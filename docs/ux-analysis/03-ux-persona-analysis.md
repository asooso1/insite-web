# insite-web UX 분석: 5가지 사용자 페르소나

> 작성일: 2026-03-23
> 범위: Phase 6 구현 완료 모듈 기준 (75% 진행도)
> 분석 대상: 목록/상세/등록 페이지 표준 패턴 + 컴포넌트

---

## 개요

insite-web은 5가지 주요 사용자 유형을 지원해야 한다. 각 페르소나별로 현재 구현된 기능과 UX 개선이 필요한 영역을 정리한다.

| 페르소나 | 특징 | 핵심 니즈 |
|---------|------|---------|
| **주니어 사용자** | 현장 신입 직원 | 직관성, 명확한 안내, 오류 복구 |
| **시니어 사용자** | 숙련 현장 직원 | 효율성, 빠른 작업, 고급 기능 |
| **현장 관리자** | 시설 관리자 | 전체 현황, 의사결정 정보, 알림 |
| **시스템 관리자** | IT 담당자 | 설정 완성도, 권한 관리, 감시 |
| **QA 테스터** | 검증 담당자 | 폼 검증, 경계 케이스, 접근성 |

---

## 1. 주니어 사용자 (현장 신입 직원) 🟦

### 사용자 프로필
- 처음 시스템을 사용하는 신입 직원
- 컴퓨터에 익숙하지 않을 수 있음
- 명확한 안내가 필수
- 에러 발생 시 불안감 큼

### 현재 구현 현황 ✅

#### 충족되는 니즈

| 기능 | 구현도 | 증거 |
|------|--------|------|
| **명확한 페이지 헤더** | 100% | `PageHeader` 컴포넌트: 제목 + 설명 + 아이콘 표시 |
| **직관적 네비게이션** | 85% | 사이드바 기반 계층 구조, 모듈별 탭 |
| **빈 상태 처리** | 90% | `EmptyState` 컴포넌트로 데이터 없음 상태 표시 |
| **간단한 액션** | 80% | 드롭다운 메뉴 기반 행 액션 (상세보기, 수정, 삭제) |
| **에러 메시지** | 75% | `EmptyState` type="error" + 재시도 버튼 |
| **새로 만들기 버튼** | 95% | PageHeader actions에 명시적 버튼 배치 |

**긍정적 사례:**
- **시설 목록 페이지** (`facilities/page.tsx`)
  ```
  PageHeader
    title: "시설 목록"
    description: "시설을 관리합니다."
    icon: Wrench
    actions: [+ 등록 버튼]
  FilterBar (선택적 필터)
  DataTable (명확한 컬럼 헤더)
  EmptyState (데이터 없음 → "첫 번째 시설을 추가해보세요")
  ```
  → 신입도 쉽게 등록하는 방법을 이해할 수 있음

- **상세 페이지의 명확한 액션**
  ```
  DropdownMenu
    상세 보기
    수정
    삭제 (위험한 액션 구분)
  ```
  → 각 액션의 의도가 명확함

#### 부족한 니즈 ❌

| 기능 | 심각도 | 이유 |
|------|--------|------|
| **온보딩 투어/튜토리얼** | 🔴 Major | 처음 사용하는 신입이 기능 발견에 어려움 |
| **인라인 도움말 (Tooltip)** | 🟡 Minor | 필터 옵션, 설정 항목의 의미가 불명확할 수 있음 |
| **필드별 에러 메시지** | 🟡 Minor | 폼 검증 실패 시 어떤 필드가 잘못되었는지 명확하지 않을 수 있음 |
| **작업 완료 확인** | 🟡 Minor | 삭제/수정 후 토스트 피드백은 있지만 페이지 새로고침 지연 |
| **컨텍스트 도움말** | 🔴 Major | 설정 페이지의 복잡한 탭들 (미화 분류, 계수 등) 설명 부족 |
| **키보드 접근** | 🟡 Minor | 마우스 의존도 높음 (Tab 키 네비게이션 미지원) |

**주요 문제점:**

1. **필터 사용법 불명확**
   - `FilterBar` 컴포넌트는 깔끔하지만, 각 필터의 용도 설명 없음
   - 예: "설비이력 있음" 필터가 정확히 뭔지 신입은 모를 수 있음

2. **폼 검증 에러 가시성 부족**
   - 필수 필드 마크(`*`)는 있지만, 검증 실패 메시지가 명확하지 않을 수 있음
   - 예: "시설명을 입력해주세요" vs 그냥 빨간 밑줄

3. **설정 페이지 탭 수가 많음**
   - 7개 탭 (기본 코드, 설비 분류, 표준 설비, 미화 4개) → 신입은 어떤 탭을 선택해야 할지 혼란
   - 탭 설명이 아이콘+텍스트인데, 도메인 지식 없으면 이해 어려움

### UX 위험 요소 (발견된 문제)

**H1. 시스템 상태 가시성 - 중간 우선순위**
- 데이터 로딩 중 스켈레톤은 있지만, 검색/필터 적용 중 상태 표시 불명확
- 페이지 전환 시 로딩 스피너 없음 → 신입이 클릭 안 됐다고 생각할 수 있음

**H2. 실제 사용 맥락과의 불일치**
- 설정 페이지의 "기본 코드" 항목들이 현장 직원 입장에서 의미 불명확
- 예: "inputTypeName" "scopeName" → 기술 용어로 인한 인지 부하

**H6. 인식 vs 회상 - 낮은 우선순위**
- 자동완성 없음 (예: 장비번호 검색 시 이전 검색어 목록 X)
- 저장된 필터 기본값 없음 (매번 필터를 다시 설정해야 함)

### 개선 제안

| # | 개선 항목 | 영향도 | 난이도 |
|---|---------|--------|--------|
| 1 | 처음 사용자용 인라인 투어 (Intro.js 등) | 높음 | 중간 |
| 2 | 필터별 Tooltip 추가 ("설비이력이 있는 시설만 표시" 등) | 중간 | 낮음 |
| 3 | 폼 필드 에러 메시지 개선 (필드명 + 구체적 이유) | 중간 | 낮음 |
| 4 | 설정 페이지 탭 분류 (관리자 권한만 표시) | 높음 | 높음 |
| 5 | 키보드 네비게이션 지원 (Tab 순서 정의) | 중간 | 중간 |

## 2. 시니어 사용자 (숙련 현장 직원) 🟩

### 사용자 프로필
- 3개월 이상 시스템 사용한 숙련 직원
- 반복 작업 (일일 보고, 작업 배정 등) 다량 처리
- 속도와 효율성 중시
- 고급 기능 활용 가능

### 현재 구현 현황 ✅

#### 충족되는 니즈

| 기능 | 구현도 | 증거 |
|------|--------|------|
| **빠른 등록** | 70% | 목록 페이지에서 `+ 등록` 버튼으로 1클릭 진입 |
| **필터 저장** | 0% | 필터 상태가 URL에 반영되지 않음 (nuqs 미적용) |
| **일괄 선택** | 0% | DataTable에 체크박스 없음 (선택 후 일괄 삭제/수정 불가) |
| **엑셀 다운로드** | 85% | facilities/users 페이지에 구현 |
| **빠른 수정** | 75% | 드롭다운 메뉴로 수정 페이지 진입 가능 |
| **데이터 복사** | 50% | 설정 > 표준 설비에만 "복사" 기능 있음 |

#### 부족한 니즈 ❌

| 기능 | 심각도 | 이유 |
|------|--------|------|
| **키보드 단축키** | 🔴 Major | 마우스 없이 빠른 작업 불가 |
| **필터 저장/즐겨찾기** | 🔴 Major | 매번 같은 필터 조건 재입력 필요 |
| **벌크 액션** | 🔴 Major | 다중 선택 후 일괄 삭제/변경 불가 |
| **검색어 자동완성** | 🟡 Minor | 이전에 입력한 검색어 목록 없음 |
| **빠른 필터 리셋** | 🟡 Minor | "초기화" 버튼은 있지만 [X] 빠른 삭제 없음 |
| **데이터 밀도** | 🔴 Major | 한 화면에 보이는 행 수 적음 (기본 10-20행) |
| **정렬 기능** | 🟡 Minor | DataTable이 정렬 불가능한 컬럼들이 있음 |

**주요 문제점:**

1. **필터 상태 미보존**
   - 사용자가 상태별로 필터링한 후 상세 페이지 진입 → 뒤로가기 시 필터 초기화됨
   - nuqs (URL 기반 상태 관리)가 일부 모듈에만 적용됨

   ```typescript
   // 현재: useState 기반 (URL에 반영 X)
   const [filters, setFilters] = useState(INITIAL_FILTERS);
   
   // 필요: nuqs 적용
   const [filters, setFilters] = useQueryState('filters', parseAsString);
   ```

2. **벌크 액션 없음**
   - 사용자가 10개 행을 선택하고 일괄 삭제하고 싶지만 불가능
   - 현재: 개별 드롭다운 메뉴에서만 삭제 가능

3. **데이터 밀도 낮음**
   - 기본 페이지 크기: 10-20행
   - 시니어 사용자가 한 번에 100행 보기를 원할 수도 있음
   - `size` 선택 드롭다운 없음

### UX 위험 요소

**H7. 유연성과 효율성 - 높은 우선순위**
- 키보드 단축키 부재 (Ctrl+N: 새로 만들기, Ctrl+Shift+F: 필터 등)
- 마우스 의존도 높음

**H3. 사용자 제어와 자유도 - 높은 우선순위**
- 필터 선택 후 변경하려면 "초기화" → 다시 선택 필요
- 각 필터에 [X] 빠른 삭제 버튼 없음

**H6. 인식 vs 회상 - 중간 우선순위**
- 자동 저장되는 기본값 없음
- 검색 히스토리 없음

### 개선 제안

| # | 개선 항목 | 영향도 | 난이도 |
|---|---------|--------|--------|
| 1 | nuqs 도입으로 필터 상태 URL 저장 | 높음 | 중간 |
| 2 | DataTable에 다중 행 선택 + 벌크 삭제/변경 | 높음 | 높음 |
| 3 | 페이지 크기 선택 드롭다운 (10/20/50/100) | 중간 | 낮음 |
| 4 | 각 필터에 [X] 빠른 삭제 버튼 | 중간 | 낮음 |
| 5 | 자주 쓰는 필터 조합 저장 기능 | 중간 | 높음 |
| 6 | 검색 자동완성 (localStorage 기반) | 낮음 | 중간 |
| 7 | 키보드 단축키 (Ctrl+N, Ctrl+E, Ctrl+D) | 낮음 | 높음 |

## 3. 현장 관리자 (시설 관리자) 🟨

### 사용자 프로필
- 빌딩/구역 전체를 관리하는 관리자
- 의사결정을 위해 전체 현황과 요약 정보 필요
- 시스템 알림/보고서로 주요 정보 파악
- 승인 워크플로우 필요

### 현재 구현 현황 ✅

#### 충족되는 니즈

| 기능 | 구현도 | 증거 |
|------|--------|------|
| **목록 조회** | 95% | 모든 모듈에 목록 페이지 구현 |
| **필터링** | 85% | 상태별, 우선순위별 필터링 |
| **검색** | 80% | 이름, 번호, 연락처 등으로 검색 |
| **엑셀 다운로드** | 70% | facilities, users에 구현 (다른 모듈은 미구현) |
| **상태 배지** | 90% | StatusBadge로 시각적 구분 |

#### 부족한 니즈 ❌

| 기능 | 심각도 | 이유 |
|------|--------|------|
| **대시보드** | 🔴 Major | 실시간 KPI, 요약 차트 없음 |
| **알림/노티피케이션** | 🔴 Major | 긴급 작업, 미승인 항목 알림 없음 |
| **통계 보고서** | 🔴 Major | 월별/분기별 성과 리포트 기능 없음 |
| **승인 워크플로우** | 🟡 Minor | 결재 진행 상태 추적 기능 없음 |
| **KPI 위젯** | 🔴 Major | 상단에 주요 지표 카드 없음 (총 작업, 진행중, 완료 등) |
| **실시간 업데이트** | 🟡 Minor | SSE/WebSocket으로 자동 갱신 없음 (수동 새로고침만 가능) |
| **역할별 대시보드** | 🔴 Major | 모든 사용자가 같은 관리자 화면 봄 |

**주요 문제점:**

1. **대시보드 부재**
   - 관리자가 현황을 한눈에 파악할 수 없음
   - 여러 모듈을 돌아다니며 수동으로 확인 필요
   - 예: "오늘 처리해야 할 긴급 작업이 몇 개인가?" → 일일이 확인해야 함

2. **KPI 정보 부족**
   - 현장 수시업무 목록에 총 작업 수, 진행중, 완료 통계가 없음
   - PageHeader.stats를 활용하면 개선 가능

   ```typescript
   // 개선안: PageHeader에 통계 배지 추가
   <PageHeader
     title="현장 수시업무"
     stats={[
       { label: "전체", value: 128 },
       { label: "진행중", value: 12 },
       { label: "완료", value: 85 },
     ]}
   />
   ```

3. **알림 기능 없음**
   - 신규 항목 생성 시 알림 없음
   - 긴급 우선순위 항목이 추가되어도 모를 수 있음

### UX 위험 요소

**H1. 시스템 상태 가시성 - 최우선**
- 관리자가 현황을 놓치기 쉬움
- 대시보드 미구현으로 의사결정 정보 부족

**H5. 에러 예방 - 높은 우선순위**
- 승인/결재 기능 부재로 잘못된 상태 변경 가능
- 예: 누구나 완료 상태로 변경 가능

**H2. 시스템과 실제 세계의 일치 - 중간 우선순위**
- "미화 분류", "미화 계수" 등 설정 항목의 도메인 용어가 관리자에게 생소할 수 있음

### 개선 제안

| # | 개선 항목 | 영향도 | 난이도 |
|---|---------|--------|--------|
| 1 | 관리자 대시보드 구현 (KPI + 실시간 차트) | 높음 | 높음 |
| 2 | 알림 센터 (긴급 항목, 미승인 대기 등) | 높음 | 높음 |
| 3 | 모든 모듈에 엑셀 다운로드 추가 | 중간 | 낮음 |
| 4 | PageHeader.stats 활용으로 한눈에 현황 파악 | 중간 | 낮음 |
| 5 | 월간/주간 리포트 자동 생성 | 중간 | 높음 |
| 6 | SSE 기반 실시간 업데이트 (optional) | 낮음 | 높음 |

## 4. 시스템 관리자 (IT 담당자) 🟦

### 사용자 프로필
- IT 담당자 또는 시스템 관리자
- 사용자 권한, 설정, 감사 로그 관리
- 시스템 안정성과 규정 준수 필요

### 현재 구현 현황 ✅

#### 충족되는 니즈

| 기능 | 구현도 | 증거 |
|------|--------|------|
| **사용자 관리** | 90% | /users 페이지: 사용자 목록, 생성, 수정, 삭제 |
| **사용자 검색** | 85% | 이름, 아이디, 휴대폰으로 검색 가능 |

   ```typescript

   // 개선안: 도메인 용어로 번역 필요
   <span className="text-xs text-muted-foreground">
     {config.inputTypeName} · {config.scopeName}
   </span>
   // → "텍스트 입력 · 전체 건물" 식으로 번역
   ```

## 결론

insite-web의 기본 UX는 견고하지만, **고급 기능과 관리자 기능에서 개선이 필요하다**. 

각 페르소나별 우선순위:
- **주니어**: 온보딩 투어, Tooltip 추가 → 학습곡선 단축
- **시니어**: nuqs, 벌크 액션, 페이지 크기 선택 → 반복 작업 효율화
- **관리자**: 대시보드, 알림, KPI → 의사결정 정보 제공
- **시스관**: 감사 로그, 권한 관리 UI → 거버넌스 강화
- **QA**: 폼 검증 표준화, Aria-label → 품질 보증

가장 높은 ROI를 기대할 수 있는 개선은:
1. **필터 상태 URL 저장 (nuqs)** - 낮은 난이도, 높은 영향도
2. **페이지별 KPI 통계** - 낮은 난이도, 중간 영향도
3. **벌크 액션** - 높은 난이도, 높은 영향도
