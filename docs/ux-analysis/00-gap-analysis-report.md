## 2. 완전 미구현 모듈 (v1에 있지만 v2에 없는 것)

### 2.1 BEMS (Building Energy Management System) - 85개 화면

**목적**: 빌딩의 전체 에너지 사용량 모니터링 및 최적화

**주요 기능**:
- 실시간 에너지 대시보드 (전력, 가스, 수도 사용량)
- 에너지 비용 분석 및 예측
- 설비별 에너지 소비 분석
- 에너지 절감 리포트 및 제안
- 시간대별/월별/연간 사용량 추이
- 피크 시간대 분석 및 알림
- 에너지 타겟 설정 및 진행률 추적
- 외부 기후 데이터 연동 (온도/습도)
- 실시간 알림 (이상 사용량)

**우선순위**: 🔴 **Very High** (에너지 관리는 BEMS의 핵심)

**예상 화면 구성**:
- 대시보드 메인 (5개 화면)
- 에너지 분석 (12개 화면: 전력/가스/수도/증기/열 각 2-3개)
- 설비별 분석 (15개 화면)
- 보고서 및 분석 (20개 화면)
- 설정 및 알림 (10개 화면)
- 통계/예측 모델 (23개 화면)

**기술적 복잡도**: 🔴 **High**
- 실시간 데이터 스트리밍 (WebSocket/SSE)
- 대용량 시계열 데이터 처리
- 차트/그래프 성능 최적화 (Recharts 또는 ECharts)
- 데이터 집계 및 집계(aggregation) 알고리즘

### 2.2 BECM (Boiler & Chiller Monitoring) - 64개 화면

**목적**: 냉난방 설비(보일러, 냉동기)의 운영 상태 모니터링 및 유지보수

**주요 기능**:
- 보일러/냉동기 실시간 운영 상태 모니터링
- 설비 온도, 압력, 유량 등 주요 지표 표시
- 설비 런타임 추적 (가동/정지/대기)
- 유지보수 일정 관리
- 에러 로그 및 알림
- 성능 분석 (효율성, 에너지 소비)
- 부하 분산 분석 (여러 설비 비교)
- 설비 용량 계획 (수명 예측)

**우선순위**: 🟡 **High** (FMS와 연계된 중요 기능)

**예상 화면 구성**:
- 모니터링 대시보드 (8개 화면)
- 개별 설비 상세 모니터링 (15개 화면)
- 유지보수 관리 (12개 화면)
- 성능 분석 (12개 화면)
- 알림 및 로그 (8개 화면)
- 설정 (9개 화면)

**기술적 복잡도**: 🟡 **Medium-High**
- 실시간 센서 데이터 수신 (MQTT/WebSocket)
- 임계값 기반 알림 로직
- 설비별 센서 매핑 (다대다 관계)
- 에러 코드 번역 및 원인 분석

### 2.3 FEMS (Fire Equipment Management System) - 50개 화면

**목적**: 소방 설비 및 자동화재탐지설비(AFDIS) 관리

**주요 기능**:
- 소방 설비 목록 관리 (스프링클러, 소화기, 비상구 등)
- 정기점검 일정 관리 및 추적
- 점검 기록 및 이력 관리
- 부적격 항목 추적 및 개선
- 소방 규정 준수 여부 확인
- 월간/년간 소방 보고서 자동 생성
- AFDIS 센서 상태 모니터링
- 소방 훈련 일정 및 기록 관리

**우선순위**: 🟡 **High** (법적 필수 관리 항목)

**예상 화면 구성**:
- 소방 설비 목록 관리 (8개 화면)
- 정기점검 스케줄 (10개 화면)
- 점검 기록 입력/조회 (12개 화면)
- AFDIS 모니터링 (8개 화면)
- 보고서 생성 (8개 화면)
- 규정 관리 (4개 화면)

**기술적 복잡도**: 🟡 **Medium**
- 점검 체크리스트 동적 생성
- 규정 버전 관리 (법령 변경 대응)
- 문서 생성 및 서명 관리
- 엑셀 템플릿 기반 보고서 생성

### 4.3 기능별 상세 설명

**1. 대시보드 - KPI 통계 카드**
- 현재: 위젯 기반 대시보드 있음, 하지만 주요 KPI 요약 카드 없음
- 필요: PageHeader.stats에 "전체 작업 / 진행중 / 완료" 같은 통계 표시
- 파일: `src/app/(modules)/dashboard/page.tsx`, `src/components/common/page-header.tsx`
- 영향: 관리자가 현황을 한눈에 파악 가능

**2. 필터 상태 URL 저장 (nuqs)**
- 현재: useState 기반 필터 (뒤로가기 시 초기화됨)
- 필요: nuqs를 이용해 필터를 URL 쿼리스트링에 저장
- 파일: `src/lib/hooks/use-*.ts` (모든 목록 페이지)
- 영향: 사용자가 필터링 후 상세 페이지 진입 → 뒤로가기 시 필터 복원

**3. 벌크 액션 (다중 선택)**
- 현재: DataTable에 체크박스 없음
- 필요: DataTable에 행 선택 기능 추가, 삭제/상태 변경 일괄 처리
- 파일: `src/components/data-display/data-table.tsx`
- 영향: 시니어 사용자의 반복 작업 효율화 (100행 선택 후 삭제 가능)

**4. 감사 로그 (Audit Log)**
- 현재: 없음
- 필요: 누가 언제 어떤 항목을 수정/삭제했는지 추적
- 파일: `src/app/api/audit-logs/route.ts`, `src/app/(modules)/settings/audit-logs/page.tsx`
- 영향: 규정 준수 (Compliance), 문제 추적 가능

**5. 알림 센터**
- 현재: 없음
- 필요: 긴급 작업, 미승인 항목 알림 (Bell 아이콘에 배지)
- 파일: `src/components/layout/header.tsx`, `src/app/api/notifications/route.ts`
- 영향: 관리자가 중요 사항을 놓치지 않음

**6. 실시간 데이터 업데이트 (SSE)**
- 현재: 페이지 수동 새로고침만 가능
- 필요: Server-Sent Events로 데이터 자동 갱신
- 파일: `src/app/api/dashboard/stream/route.ts`, `src/lib/hooks/use-sse.ts`
- 영향: 대시보드가 실시간 정보 제공

**7-13**: 기존 모듈 기능 추가 (낮은-중간 복잡도)

**14-17**: BEMS/BECM/FEMS 새 모듈 (높은 복잡도, Phase 10-11)

## 6. 사용자 페르소나별 핵심 미충족 니즈

### 6.1 주니어 사용자 (현장 신입)

**핵심 니즈**: 직관성, 명확한 안내, 오류 복구

**Top 3 미충족 항목**:
1. **온보딩 투어 부재** (Impact: High)
   - 문제: 처음 사용자가 기능 발견에 어려움
   - 해결: Intro.js/Joyride로 step-by-step 투어 추가
   - 예상 공수: 1-2주

2. **필터 Tooltip 없음** (Impact: Medium)
   - 문제: "설비이력 있음" 필터의 의미가 불명확
   - 해결: 각 필터에 마우스 오버 설명 추가
   - 예상 공수: 3-5일

3. **폼 에러 메시지 구체성 부족** (Impact: Medium)
   - 문제: 검증 실패 이유가 명확하지 않음
   - 해결: 필드명 + 구체적 이유 + 해결책 제시
   - 예상 공수: 5-10일

**추천 우선순위**: P1 (빠른 학습곡선 = 생산성 향상)

---

### 6.2 시니어 사용자 (숙련 현장 직원)

**핵심 니즈**: 효율성, 빠른 작업, 반복 업무 최소화

**Top 3 미충족 항목**:
1. **필터 상태 미보존** (Impact: Very High)
   - 문제: 필터링 후 상세 페이지 진입 → 뒤로가기 시 필터 초기화
   - 해결: nuqs로 필터를 URL에 저장
   - 예상 공수: 1-2주
   - 코드 예시:
   ```typescript
   // 현재 (미보존)
   const [filters, setFilters] = useState(INITIAL);
   
   // 필요 (보존)
   const [filters, setFilters] = useQueryState('filters', parseAsString);
   ```

2. **벌크 액션 없음** (Impact: Very High)
   - 문제: 10개 항목을 삭제하려면 하나하나 클릭해야 함
   - 해결: DataTable에 체크박스 추가, 일괄 삭제/수정 구현
   - 예상 공수: 2-3주
   - 영향: 반복 작업 시간 50% 단축

3. **데이터 밀도 낮음** (Impact: High)
   - 문제: 기본 페이지 크기가 10-20행으로 고정
   - 해결: 페이지 크기 선택 드롭다운 (10/20/50/100)
   - 예상 공수: 2-3일

**추천 우선순위**: P1 (빠른 ROI, 중간 난이도)

### 6.3 현장 관리자 (시설 관리자)

**핵심 니즈**: 전체 현황, 의사결정 정보, 실시간 알림

**Top 3 미충족 항목**:
1. **대시보드 부재 또는 미흡** (Impact: Very High)
   - 문제: 의사결정을 위한 요약 정보가 없음
   - 필요:
     - 상단에 KPI 카드 (오늘 처리할 작업, 진행중, 완료)
     - 실시간 차트 (시간대별 작업 현황)
     - 우선순위 작업 목록
   - 예상 공수: 2-3주
   - 파일: `src/app/(modules)/dashboard/page.tsx`

2. **알림 기능 없음** (Impact: Very High)
   - 문제: 긴급 작업이나 미승인 항목이 추가되어도 모를 수 있음
   - 해결:
     - 알림 센터 (Header의 Bell 아이콘)
     - 긴급 우선순위 항목 실시간 알림
     - 이메일/SMS 선택적 알림
   - 예상 공수: 2-3주

3. **SSE 실시간 업데이트 없음** (Impact: High)
   - 문제: 수동 새로고침만 가능
   - 해결: Server-Sent Events로 대시보드 자동 갱신
   - 예상 공수: 2-3주

**추천 우선순위**: P2 (높은 영향도, 높은 난이도)

**Top 3 미충족 항목**:

   - 예상 공수: 1-2주
   - 파일: `src/app/(modules)/users/[id]/edit/page.tsx`

3. **시스템 로그 조회 불가** (Impact: High)
   - 문제: 에러 로그, 접근 로그 조회 방법 없음
   - 해결: 시스템 로그 조회 페이지 (에러/접근/변경)
   - 예상 공수: 2주

## 11. 결론 및 권장사항

### 11.1 현황 평가

**강점**:
1. 기본 CRUD 모듈이 75% 구현되어 있음
2. 디자인 시스템과 컴포넌트 패턴이 일관성 있음
3. 인증 및 권한 체계가 확립됨
4. 모바일 반응형 대응이 완료됨

**약점**:
1. 관리자 기능 (대시보드, 알림) 미흡
2. 고급 기능 (벌크 액션, 필터 저장) 없음
3. 감시/규정 준수 기능 부재
4. 새로운 모듈 (BEMS/BECM/FEMS) 미구현

**기회**:
1. Phase A-B로 빠른 win 확보 (3-4주)
2. Phase C로 기업 거버넌스 강화
3. Phase D로 차별화된 기능 제공

---

### 11.2 권장 실행 전략

#### 우선순위 1: Phase A 완료 (3주)
- **목표**: 기본 UX 개선으로 사용성 75% → 85% 향상
- **공수**: 1명 (또는 2명 병렬)
- **예상 효과**: 모든 페르소나 만족도 +5-10%
- **리스크**: 낮음

#### 우선순위 2: Phase B 착수 (4주)
- **목표**: nuqs + 벌크 액션으로 시니어 효율성 30% 향상
- **공수**: 1-2명
- **예상 효과**: 반복 작업 시간 50% 단축
- **리스크**: 중간 (성능 테스트 필요)

#### 우선순위 3: Phase C 병렬 (8주)
- **목표**: 관리자 대시보드 + 감시 로그로 거버넌스 강화
- **공수**: 2-3명
- **예상 효과**: 관리자 의사결정 정보 제공, 규정 준수
- **리스크**: 높음 (복잡도 높음)

#### 미래 검토: Phase D (3개월+)
- **조건**: Phase C 완료 + 요구사항 명확화
- **목표**: BEMS/BECM/FEMS로 에너지/설비 관리 영역 확대
- **공수**: 2-3명 + 데이터 엔지니어
- **리스크**: 매우 높음 (새로운 도메인)

---

### 11.3 품질 보증 계획

| 활동 | 시기 | 담당 | 범위 |
|------|------|------|------|
| **유닛 테스트** | 각 Phase 중 | 개발자 | 새로운 함수/훅 80% 이상 |
| **통합 테스트** | 각 Phase 후 | QA 또는 개발자 | API 연동, 폼 검증 |
| **E2E 테스트** | Phase A, B, C | QA 테스터 | 주요 워크플로우 |
| **접근성 감사** | Phase A, C | QA 또는 WCAG 전문가 | WCAG 2.1 AA 기준 |
| **성능 테스트** | Phase B, C | 성능 테스터 | nuqs, 벌크 액션, SSE |
| **보안 검토** | Phase C 전 | 보안 담당자 | 감시 로그 암호화, CORS |
| **사용자 테스트** | Phase D 전 | UX 리서처 + 사용자 | BEMS 요구사항 검증 |

---

### 11.4 최종 결론

**insite-web은 기본 기능은 75% 완성되었지만, 고급 기능과 관리자 기능에서 개선이 필요하다.**

**가장 높은 ROI를 기대할 수 있는 개선순서**:
1. **Phase A** (3주): 모든 페르소나 만족도 향상
2. **Phase B** (4주): 시니어 사용자 효율화
3. **Phase C** (8주): 관리자 기능 강화
4. **Phase D** (3개월+): 새로운 영역 확대

**추천 일정**: 2026년 Q2-Q3에 Phase A-B-C를 순차 진행, Q4부터 Phase D 준비
